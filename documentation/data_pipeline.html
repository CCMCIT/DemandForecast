Flow — process_file(fileId)

Two committed phases. Chips show tables written (green) / read (grey).

Prepare
1

Resolve the source

get_processor(FileTypeId) → GPA's repository + map_row. Rejects a file already fully processed (status 4). Everything after is source-agnostic.

2

Read raw rows

Load all rows of this file — one raw row per voyage.

GpaFileDetail_tbl · read
3

Map in memory

map_row → MappedVoyage = header + details (container columns) + fields (build_fields: the 6 descriptive columns, trimmed, blanks skipped). No DB yet.

Phase 1 · commit → status 3
4

Insert voyages

One Voyage_tbl row per voyage (existing ones replaced in place → history). Committed first so VoyageId exists for everything below.

Voyage_tbl · write
Phase 2 · one commit, all-or-none → status 4
5

Write details

Unpivot each voyage's container columns into detail rows. Mode/Direction resolved from preloaded dicts — no per-row query.

VoyageDetails_tbl · writeMode · Direction · read
6

Write fields (get-or-create, in order)

Per value: ① FieldValue (reuse if it exists) → ② FieldTypeValue for the (type, value) pair → ③ link the voyage in VoyageFieldMap. A voyage's old maps are deleted first, then re-inserted.

① FieldValue_tbl② FieldTypeValue_tbl③ VoyageFieldMap_tbl
After
7

Classify fallen-off voyages

Voyages still “to call” but absent from this file → set Called or Cancelled from their dates.

Voyage_tbl · status update

Order in step 6 is mandatory — each row needs the id created by the previous one. Details + fields share Phase 2's single transaction, so an interruption leaves neither and the file safely re-runs from status 3.
