The code structure flow is this

The pipeline runs in two phases, and each phase is a straight line: read a file, hand rows to a repository, let the repository write the tables. Nothing in the middle touches the database, and nothing at the edges knows SQL. A file's type is decided once, by its filename prefix (GPA), and that choice selects every handler for the rest of the run.

Each layer has exactly one job, and layers only depend inward:

  • readers parse a file into list[dict] — no database.
  • repositories are the only place that reads or writes a table.
  • ingestion / processing orchestrate a run but hold no SQL.
  • mapping is the only place that knows GPA's columns.

A file's progress is tracked by a single LoadStatusId on its File_tbl row. It is also the resume marker: processing writes voyages first (status 3), then details (status 4), so an interrupted run picks up exactly where it stopped.

02

How it does it, layer by layer

The reader turns a file into plain dictionaries and normalises the headers. It never coerces types and never touches the database — that keeps it trivial to swap CSV for Excel later.

The loader takes those raw dicts, coerces each value once ("20250918" → a real date, blanks → None), builds model objects, and hands them to a repository. All DB access goes through that repository — the loader itself writes no SQL.

The models reflect tables that already exist in the database. The code never issues CREATE or ALTER — a model is a typed Python view of a live table, nothing more.

The mapper is the one GPA-aware piece of processing. An explicit table — no name-sniffing — says what each container column means, and the unpivot loop turns one wide staging row into many detail rows, skipping empty columns.

The writer consumes that source-agnostic result and persists it through the voyage repositories, in two steps so the runner can commit them separately.

The runner owns the transaction. It writes every voyage, commits, marks the file at status 3, then writes every detail in a single commit and marks status 4. A user stop is caught apart from a real failure — the former resumes, the latter is logged and marked ERROR.

04

Why models & repositories, not pure SQL inserts

The same File insert could be a raw INSERT. Follow that path step by step with our example and the hidden costs surface. Here is the hand-written version:

1
Identity is handed to you.

add() flushes and fills file.FileId in Python, so the staging rows and voyages reference it directly. In raw SQL you juggle SCOPE_IDENTITY() / OUTPUT and thread the id by hand across every dependent table.

2
Type coercion lives in one place.

The loader converts "20250918" to a real date and blanks to None once, in _date / _int. Raw SQL scatters formatting across every statement and string-builds values — which is also where SQL injection creeps in.

3
One unit of work, one transaction.

The session commits the File row and all its details together, or rolls back and records an ERROR row. That "commit all or fail cleanly" rule is written once in the runner, not repeated as manual BEGIN TRAN blocks.

4
Parameters are bound, not interpolated.

Every value goes to the driver as a bound parameter. A voyage code with an apostrophe, or an odd terminal name, can never break — or rewrite — a statement.

5
New file types cost no core changes.

Adding NCSPA means a new reader, loader, and mapper registered in a dictionary. The runner, writer, and base repository are reused untouched — the shape that raw, per-table SQL cannot give you without copy-paste.

6
Business rules stay centralised.

Reprocessing a voyage — replace it, archive the old version to history, re-insert details — lives once in _replace_or_add_voyage. In raw SQL that check-update-delete-reinsert dance would be duplicated at every call site.

7
The schema has one source of truth.

Models reflect the live tables, so a renamed column is fixed in one model, not hunted through dozens of scattered SQL strings. Database-first still holds: the code reflects the schema, it never creates or alters it.

The trade in one line: raw SQL is fewer characters for the very first insert, and more characters — plus more risk — for every insert, table, and file type after it. Models and repositories pay a small upfront cost to make each following step boring and safe.
05

Models & inserts vs stored procedures

Both can write the same rows. The real question is where the logic lives — in the Python app, or compiled inside SQL Server. This pipeline starts by reading a file, which a stored procedure cannot do, so the balance tips before the first insert.

Models + repositories · app-side

Strong at:

  • parsing files (CSV/Excel) and coercing types — only the app can do this
  • the branching unpivot / mapping rules, in readable Python
  • unit tests with pytest, version control, code review
  • one deployable; new file type = new classes, no core change

Weak at:

  • row-by-row ORM writes are slower than set-based SQL; more round-trips
Stored procedures · DB-side

Strong at:

  • set-based bulk insert/update, run closest to the data — fastest at volume
  • one copy of logic reusable by any app hitting the database
  • tighter security: grant EXECUTE instead of table access

Weak at:

  • cannot read files — you still need app code to parse them
  • T-SQL is harder to test, diff, and review; branching rules get clumsy
  • logic gets locked inside the database
Better for this project: models + repositories. The entry point is a file on disk and the hard part is conditional mapping — both belong in Python, tested and versioned. Stored procedures still win at one thing: heavy set-based bulk writes. So the sensible hybrid is to keep parsing and rules in the app, and — only if insert volume ever hurts — push the bulk write into a procedure. Database-first still holds: a procedure, like a table, would be owned by the DB, never created by this code.
